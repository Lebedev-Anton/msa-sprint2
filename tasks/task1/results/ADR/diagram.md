### <a name="_qmphm5d6rvi3"></a>**Решение**
#### Обоснование решения

**Почему BookingService:**

1. **Наибольшая нагрузка** — каждое создание бронирования вызывает 4 других сервиса, что делает его главным bottleneck при пиковых нагрузках
2. **Чёткий ограниченный контекст** — входы/выходы определены REST-эндпоинтами, бизнес-логика самодостаточна
3. **Зависимости только на чтение** — BookingService читает данные других сервисов, но не владеет ими, что упрощает миграцию
4. **Бизнес-ценность** — бронирование является основной revenue-generating функцией платформы

**Выбор технологий:**

- **gRPC** для межсервисного взаимодействия — типобезопасность, производительность, поддержка streaming в будущем
- **API Gateway** для маршрутизации — позволяет постепенно перенаправлять трафик с монолита на новый сервис



### <a name="_bjrr7veeh80c"></a>**Альтернативы**
Опишите здесь наиболее важные альтернативные решения.

#### Альтернатива 1: Начать с UserService

UserService проще и имеет меньше зависимостей.

**Почему отвергнута:**
- Не решает главную проблему — пиковые нагрузки на бронирование

#### Альтернатива 2: Выделить ReviewService первым

ReviewService имеет проблему производительности (загрузка всех отзывов в память).

**Почему отвергнута:**
- ReviewService — вспомогательный, не самостоятельный юзкейс
- Не имеет прямого влияния на revenue
- Лучше оптимизировать в составе монолита или выделить вместе с BookingService

**Недостатки, ограничения, риски**

Подробно опишите здесь недостатки, ограничения и риски выбранного решения.

| **Риск** | **Описание** |
|---|---|
| Консистентность данных | При разделении БД возможны рассинхронизации между Booking DB и Main DB | 
| Регрессия производительности | Новый сервис может работать медленнее монолита из-за сетевых вызовов |
| Каскадные отказы | При падении монолита BookingService не сможет валидировать |
| Сложность отладки | Распределённая система сложнее в диагностике |
| Дублирование логики | В период миграции логика дублируется в монолите и сервисе |
| Стоимость инфраструктуры | Отдельный сервис + БД + мониторинг увеличивают затраты |
| Командная координация | Команды монолита и BookingService должны согласовывать API-контракты |
